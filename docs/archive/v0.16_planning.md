## From the audit (recommendation 7)

### D1 — Phase 4 needs a single-valued-predicate guard in the plan

Phase 4 step 7 of the v0.15 plan says "matching predicate but contradicting
value → contradicted" with no guard. That instruction is wrong for multi-valued
KB properties (P39 position-held, P106 occupation, P166 award): the KB simply
holds *other* values and the claim's value may also be true. The implementation
inherited the bug (audit M4).

**v0.16 plan change.** Phase 4's KB-verifier step must specify: a value mismatch
yields `contradicted` only for a *functional* (single-valued) predicate, else
`no_match`. The predicate-translation oracle must carry a `single_valued` field
(the fix-up added this column — see D4) and the plan must describe generating it.

### D2 — Phase 6 needs a non-mocked-substrate derivation-test floor

The deferred-calibration variant let the derivation walker reach
`phase-10-complete` with its core capability (distribution-gated subsumption
traversal) stubbed as `pass`. Mocked unit tests with gate-closing mocks
(`MockKB.subsumption` always `unrelated`, `MockTransport` distribution always
`neither`) were too weak a proxy — they could not have failed.

**v0.16 plan change.** Phase 6's acceptance criteria must require, as a hard
floor, **at least one integration test per failure mode that walks an actual
chain against seeded substrate rows** (not gate-closing mocks). The per-phase
test-count target for substrate/walker phases should be treated as a floor, not
a soft target. (The fix-up added `test_walker_failure_modes.py` as the model.)

### D3 — Run log must quote per-phase targets verbatim

The Phase 6 run-log entry stated "target was ~50 new" against an actual plan
target of ~80, making a 51%-under-target shortfall look like 22%-under.

**v0.16 plan change.** The run-log template must require the per-phase test
target to be quoted verbatim from the implementation plan, with the phase plan
file and line cited. (The fix-up corrected the Phase 6 entry.)

## Observations from the fix-up

### D4 — Architecture §5.2 schema should include `single_valued`

The fix-up added a `single_valued INTEGER DEFAULT 0` column to the
`predicate_translation` table (user-authorized) to support D1's guard. The
architecture document §5.2 prints the `predicate_translation` schema and does
**not** include this column. The architecture doc should be updated to add
`single_valued` so the schema and the implementation agree.

**Update (fixup-2).** Fix-up 2 backfilled `single_valued` into all 61 seed
entries and added the field to `load_seeds.py` and the seed-file format. 11
predicates are functional (`single_valued = 1`): `born_in`, `died_in`,
`born_on`, `died_on`, `capital_of`, `has_capital`, `continent_of`,
`founded_in_year`, `head_of_government`, `head_of_state`, `gender`. All others
are multi-valued — the conservative default, since a wrong `1` produces false
*contradictions* while a wrong `0` produces only false *abstains* (the accepted
§3.2 cost). The re-audit's functional list also named `country_of` and
`mother_of`; fix-up 2 reclassified both to multi-valued — `country_of` because
the predicate name is not subject-type-constrained (citizenship is plural), and
`mother_of` because its seed `slot_to_qualifier` is inverted (the Aedos subject
is the mother, who has many children). The §5.2 schema update should also note
that the seed-file format carries `single_valued`. Per-predicate reasoning is
in `docs/v0.15_build_log/fixup2_report.md`.

**Resolved (Phase A cleanup, `9b276bd`).** Architecture §5.2 now prints the
`single_valued` column and documents its semantics (1 = functional, 0 =
multi-valued; conservative default 0) plus a note that the seed-file format
carries the field, referencing `fixup2_report.md` for the per-predicate
classification.

**Resolved (Phase B, B1 — `6392fa4`).** `database.py`'s `CREATE TABLE`
`predicate_translation` block now writes `single_valued INTEGER NOT NULL
DEFAULT 0`, matching the N6 `ALTER TABLE` migration and architecture §5.2. The
two were inconsistent — a fresh database got the nullable form, a migrated one
the non-nullable form (the schema-consistency item Phase A's report flagged).

### D5 — The walker has no KB-sourced neighbor *enumeration*

The fix-up implemented the walker's subsumption traversal by enumerating
neighbors from substrate `subsumption` rows (`SubsumptionOracle.find_neighbors`).
The KB protocol's three operations (`resolve_entity`, `lookup_statements`,
`subsumption`) all check or fetch for a *known* entity/pair — none *enumerates*
an entity's taxonomy neighbors. So in a cold-start system with no seeded
subsumption rows, the walker cannot discover a KB-sourced part_of/is_a chain on
its own; the substrate must first be populated (by seeds or prior `consult`
calls). v0.16 should either add a neighbor-enumeration protocol operation or
specify how the substrate's subsumption rows get populated ahead of a walk.

### D6 — Retraction propagation is session-local

`RetractionPropagator`'s verdict-trace index is in-memory and rebuilt per
process. The fix-up wired `record_verdict_trace` (from the aggregator) and the
`retracted_at` UPDATE (from `ContradictionTracer`), so propagation works within
a process, but it does not survive across verification runs. v0.16 should add a
persistent `verdict_traces` table (or audit-log replay on startup) so
architecture §7.3's over-time soundness holds across process restarts.

**Resolved (Phase B, B4 — `70da183`).** `RetractionPropagator.replay()`
rehydrates the in-memory trace index from persisted `verdict_recorded` audit
events at process startup — **Option β (audit-log replay)**, chosen over a new
`verdict_traces` table because the audit log already carries `source_rows`, so
β adds only a consumer and no table to keep consistent. `build_pipeline` calls
`replay()`; `ContradictionTracer` replays its fallback propagator;
`propagate_retraction` is unchanged. Over-time soundness (§7.3) now holds across
process restarts. The retraction *cascade* and re-derivation (D14) remain
deferred. Decision rationale: `docs/phase_B_report.md`.

### D7 — The `predicate_equivalence` walker edge is largely redundant

`Walker._expand_via_substrate` emits `predicate_equivalence` edges via
`predicate_translation.query_neighbors`, but `TierU.lookup` stage 3 already
broadens by the same oracle, and an equivalent predicate shares the same
`kb_property` so its KB lookup is identical to the original's. The edge rarely
changes an outcome. v0.16 should either remove it or give it a distinct job.

**Resolved (Phase A cleanup, `8031d04`).** The `predicate_equivalence` edge was
removed from `Walker._expand_via_substrate`. `query_neighbors` stays —
`TierU._stage3` still uses it. Stash-and-verify confirmed no test required the
edge: the suite passes identically with it removed and restored.

### D8 — Audit-logging interface is inconsistent

`tier_u.py` and the oracles log via `log_event(conn, ...)` (a function);
`consistency.py`, `retraction.py`, and `contradiction_tracer.py` call
`self._audit.log(...)` (a method) but `audit/log.py` exposes no object with a
`.log` method — those branches are dead (audit is always `None` there). v0.16
should unify on one audit-logging interface.

**Resolved (Phase A cleanup, `cefe65f`).** `consistency.py`, `retraction.py`
and `contradiction_tracer.py` were standardized on the `log_event(conn, ...)`
function form; the dead `audit_log` constructor parameter was removed from all
three. The events involved (`consistency_violation`,
`circuit_breaker_triggered`, `verdict_retracted`, `contradiction_traced`) are
the events architecture §5.4 says the audit log records, so the fix makes
required logging real. Classified additive for the three classes
`build_pipeline` wires with a db; `contradiction_traced` is additive only
where a db is supplied, since `ContradictionTracer` is still unwired in the
deployed pipeline (D15).

### D9 — `VerificationResult.audit_log_entries` needs verification_context plumbing

The fix-up un-stubbed `audit_log_entries` (m6) by having the aggregator log a
`verdict_recorded` event per claim and collecting those ids. A complete
implementation — referencing *every* audit event created during a verification
(oracle `row_created`, `budget_exceeded`, `consistency_violation`, …) — needs a
verification id threaded through `log_event`'s `verification_context` argument
at every call site. v0.16 should specify that plumbing.

### D10 — Tier U → Python composition is unsupported

The walker cannot feed a Tier U value into a Python computation (the
`der_cross_007` "Asa's birth year plus 30 is 2003" pattern): `PythonVerifier`
sees only the claim's own slots, not premises retrieved from Tier U. v0.16
should decide whether cross-source chains into Python are in scope and, if so,
specify how the walker supplies computed-claim inputs.

### D11 — Corpus completeness (audit m1)

`extraction_corpus.jsonl` has 57 cases against a Phase 1 "≥ 60" requirement.
v0.16's plan should require a corpus-count assertion in each phase's acceptance
criteria so a short corpus is caught at build time. (Deferred this session as an
unrelated-file minor finding.)

## From the re-audit (`reaudit_report.md` recommendation 3)

These were identified by the re-audit and are recorded here for the v0.16 plan.
D12 and D17 carry fixup-2 updates noting code-level resolution.

### D12 — Architecture §5.4 consistency rule vs. inverse predicates

The `transitive_equivalence_violation` rule flags two predicates mapped to the
same KB property with different `slot_to_qualifier` as a conflict, but inverse
predicates (`capital_of`/`has_capital`, both on P36) legitimately do exactly
that (re-audit N5). The rule, or the seed-pack representation of inverse
predicates, must be revised.

**Update (fixup-2).** N5 is **fixed in code** — `consistency.py`'s rule is now
direction-aware (`_is_inverse_mapping`): two predicates whose `slot_to_qualifier`
maps are exact subject/object inversions of each other are compatible inverses,
not a conflict. What remains for v0.16 is the **architecture §5.4 wording**: the
document still describes `transitive_equivalence_violation` as direction-blind
and should be revised to state the inverse-predicate exemption.

**Resolved (Phase A cleanup, `9b276bd`).** Architecture §5.4's
`transitive_equivalence_violation` bullet now states the inverse-predicate
exemption: two predicates on the same KB property whose `slot_to_qualifier`
maps are exact subject/object inversions are not a conflict; any other
divergence remains one.

### D13 — Retraction propagation must cover KB-grounded verdicts (re-audit N4)

Trace edges for KB premise lookups carry no retractable identifier, and the
`entity_resolution_cache` (which *is* retractable) is never referenced by any
trace edge. A purely KB-grounded verdict records empty `source_rows` and can
never be reached by `propagate_retraction`. Architecture §7.3's over-time
soundness is unreachable for the most common verdict type. v0.16 should record
the `entity_resolution_cache` row id(s) on the KB `premise_lookup` edge, add
`entity_resolution_cache` to the aggregator's `_TRACE_ROW_ID_KEYS`, and decide
how a cached `lookup_statements` result is identified for retraction.

### D14 — The retraction cascade and re-derivation (re-audit M2)

Architecture §7.3's "marked for re-derivation … may be re-derived from
remaining premises" and the verdict→dependent-verdict cascade are unimplemented;
`propagate_retraction` is a single row→verdict hop whose return value is
discarded by `resolve_conflict` and `trace_contradiction`. v0.16 should
implement the cascade (a retracted verdict's own consequences become a new
retraction event) and re-derivation, or explicitly scope re-derivation out in
the architecture.

### D15 — `ContradictionTracer` is not wired into `app.py`

The fix-up (fix-up 1) wired `ConsistencyChecker` and `RetractionPropagator` into
the `/chat` pipeline but not `ContradictionTracer`; it is constructed only in
tests. Downstream contradiction tracing (architecture §7.3 retraction source #2)
is inert in the deployed pipeline and has no trigger. v0.16 should construct
`ContradictionTracer` in the pipeline and define its trigger (a deployment
feedback path, or automatic re-checking of verdicts against later premises).

### D16 — Object-conflict belief revision

The walker's belief revision catches polarity contradictions only. With
`single_valued` now populated (D1/D4), a functional predicate's *object*
conflict (`Asa lives_in NYC` in Tier U vs a claimed `Asa lives_in Boston`)
could also be detected. Relatedly, the Tier U write-path's contradiction-closure
(`tier_u.py`) closes a prior row on *any* object difference, which is wrong for
multi-valued predicates and should consult `single_valued`.

**Resolved (Phase B, B2/B3 — `662332b`, `a7d26e4`).** The walker now detects
object conflicts on functional predicates: a positive claim whose functional
predicate has a different currently-valid Tier U value is `contradicted`
(trace marker `belief_revision: object_conflict`). The Tier U write path closes
a prior row only on a genuine contradiction — same object + opposite polarity,
or a functional predicate's different object at the same positive polarity;
multi-valued differences (and contrastive corrections) write parallel rows. The
negated-claim direction — verifying a negation that a functional prior implies —
was deliberately left conservative (abstain), see `docs/phase_B_report.md`
Decision 1. Two scoped limitations remain for v0.16: `lookup_object_conflict` is
literal-match only (no cross-predicate / cross-entity broadening), and the
architecture wording lags the code (see D21).

**Supplemental finding (Phase H, 2026-05-23).** Phase E v2 reported 2
universal false-verifieds on `der_revision_001` / `der_revision_002`
across all four candidates and attributed the gap to "the walker's
belief-revision integration not consulting the updated cardinality
classification correctly." The original Phase H prompt opened D16 as a
walker bug to fix. The Phase H D16 diagnostic
(`scripts/diag_d16.py` + `docs/phase_H/d16_fix.md`) **refutes that
attribution**: the walker code is correct. The two false-verifieds
trace to **cross-case Tier U state leakage in the calibration
harness**. `der_cross_002` writes `(Asa, employed_by, Google, +1)` to
the shared in-memory DB; `der_cross_009` writes `(Asa, prefers,
coffee, +1)`. Both rows persist into the later `der_revision_*` cases
because the harness shares one DB across all 50 corpus cases. The
walker's Stage 1 literal lookup then hits the leaked row and returns
`verified` — soundly, given its inputs. The "soundness violation"
framing of the original finding is incorrect; the harness had a
state-isolation defect (now fixed in Phase H), and the walker has been
correct throughout. Phase E v2's accuracy numbers across all corpora
whose runners mutate case-specific state were measured imprecisely;
the broader audit is captured as D50 below.

### D17 — Threshold single-source-of-truth (re-audit N7) + schema migration (N6)

Single source of truth for calibration thresholds, and a migration strategy for
schema columns.

**Update (fixup-2).** Both are **resolved**. N6 — `database.py`'s `create_schema`
now runs an idempotent `ALTER TABLE … ADD COLUMN single_valued` migration guard.
N7 — `THRESHOLDS` in `test_corpus_runner.py` is the documented single source of
truth, and `tests/unit/test_runbook_thresholds.py` fails CI if the
runbook's Step 4 threshold table diverges from it. No v0.16 work remains for D17.

## From fixup-2 (noticed in passing — not in fixup-2's scope)

### D18 — The chat-wrapper never extracts claims (stale `extract` signature)

`ChatWrapper.respond` (`chat_wrapper.py`) calls
`self._extractor.extract(draft, asserting_party=asserting_party)`, but
`Extractor.extract`'s signature is `extract(text, context: ExtractionContext)`
— the `asserting_party=` keyword raises `TypeError`, which `respond`'s broad
`except Exception: claims = []` swallows. So the deployed `/chat` path extracts
**zero claims** and verifies nothing; every response is pass-through. This was
noticed while extracting the shared pipeline for Cluster B (the benchmark's
`AedosRunner` had the same stale-signature bug, fixed there under M5). It does
**not** block Phase 10.5 — the calibration corpora and the medium-bar
evaluation run through the runner/walker directly, and the deployment layer is
not part of the architecture (§4.6) — but it makes the chat-wrapper deployment
verification-inert. v0.16 should fix the call to pass an `ExtractionContext`
and add an end-to-end `/chat` test that asserts claims are actually extracted.

**Update (fixup-3).** Confirmed **out of Phase 10.5 scope** by fix-up 3's scope
check 2 (`docs/v0.15_build_log/fixup3_scope.md`): `benchmark.py`'s `AedosRunner.run_case`
builds the verification pipeline via `build_pipeline` and calls
`extractor.extract` / `walker.walk` / `aggregator.aggregate` directly
(`benchmark.py:189,194,203-204`) — it never routes through `/chat`. The
calibration corpus runner is likewise direct. So D18 does not distort Phase
10.5's measurement. It remains v0.16-scope: the `/chat` chat-wrapper deployment
is still verification-inert until the `ExtractionContext` signature is fixed.

**Resolved (Phase A cleanup, `e2c8d45`).** `ChatWrapper.respond` now builds an
`ExtractionContext` (`context_type="chat_user"`) and passes it positionally.
The broad `except Exception: claims = []` was removed entirely — it was the
construct that hid the bug. `tests/integration/test_chat_wrapper.py` adds an
end-to-end test asserting claims are extracted through `/chat` (stash-and-verify
against `v0.15.0-rc.1` confirms it fails pre-fix) and a response-shape test.

### D19 — The KB verifier ignores `slot_to_qualifier`; inverse predicates look up the wrong entity

`KBVerifier.verify` resolves `claim.subject` and calls
`lookup_statements(subject_id, meta.kb_property)` — it never consults
`meta.slot_to_qualifier`. For an inverse predicate whose seed maps the Aedos
*subject* to the KB `statement_value` rather than `statement_subject`
(`capital_of` on P36, `mother_of` on P25), the KB stores the statement on the
*object* entity, so the lookup queries the wrong entity and returns nothing →
`NO_MATCH`. Noticed while classifying `single_valued` for the inverse seed
predicates (D4). The KB path for every inverse-direction seed predicate is
therefore inert. v0.16 should make the KB verifier honor `slot_to_qualifier`'s
subject/object direction (resolve and look up against the entity the KB
actually keys the statement on).

**Update (fixup-3).** **Resolved.** Fix-up 3's Cluster D19 (`6a6a466`) added
`_lookup_targets` to `kb_verifier.py`: `verify` now reads `slot_to_qualifier`,
keys the lookup on whichever entity the KB stores the statement on (the claim's
object for an inverse predicate), and compares against the other slot. The two
inverse seed predicates (`capital_of`, `mother_of`) produce real verdicts; the
trace records the direction as `lookup_inverted`. An uninterpretable
`slot_to_qualifier` abstains with `NO_KB_PATH` rather than crashing. Phase 10.5
`kb_mapping`/`derivation` cases over `capital_of`/`mother_of` no longer abstain
for this reason. What remains for v0.16 is the **architecture wording**: §5.2 /
§6.2 should state that `slot_to_qualifier` governs the KB lookup direction (see
D20). Per-predicate detail in `docs/v0.15_build_log/fixup3_report.md`.

### D20 — KB verifier trace field names and architecture lookup-direction wording

Fix-up 3 (D19) made the KB verifier's lookup direction depend on
`slot_to_qualifier`. Two follow-ups for v0.16, both cosmetic / documentation —
neither blocks anything:

1. **Trace field names.** `KBVerdict.trace` keeps the field names
   `object_resolved`, `object_value`, and the abstention reasons
   `subject_unresolved` / `object_unresolved`. Read as KB-*statement*-relative
   (statement subject / statement value) they are correct under both lookup
   directions, and the `lookup_inverted` flag disambiguates; fix-up 3 kept them
   to avoid churning the four fixup-2 N1 tests and the walker. A future audit
   may prefer direction-neutral names (`value_resolved`,
   `lookup_subject_unresolved`, …). If renamed, every trace consumer must be
   updated together.
2. **Architecture wording.** The architecture §5.2 (predicate_translation
   schema) and §6.2 (`lookup_statements`) describe the KB lookup as keyed on the
   claim subject. They should state that `slot_to_qualifier`'s subject/object
   direction governs which entity the lookup is keyed on, so the document and
   the D19 implementation agree.

**Resolved (Phase A cleanup).** Part 2 (architecture wording) is resolved in
`9b276bd`: §5.2 now documents that `slot_to_qualifier`'s `subject`/`object`
keys govern KB lookup direction, and §6.2 states the lookup is keyed on
whichever Aedos slot maps to the KB statement subject (the claim's subject or
object). Part 1 (the cosmetic `KBVerdict.trace` field-name rename) was
addressed in the release-prep session's R2 and is not revisited here. No D20
work remains.

### D21 — Architecture wording lags object-conflict belief revision

D16 added object-conflict belief revision: the walker (§6.4) now contradicts a
positive claim whose functional predicate holds a different currently-valid
Tier U value, and the Tier U write path (§6.1) closes a prior row on a
functional object conflict. The architecture document still describes only
*polarity*-based contradiction — §6.1's write-path prose says contradictions
are "asserted X, now asserts not-X", and §6.4 does not mention object-conflict
revision. Per Phase B's scope rule the architecture was not edited for this
capability change. v0.16 should update §6.1 and §6.4 to describe object-conflict
belief revision and the functional-predicate write-path closure rule (the
architecture-wording-lag pattern already seen in D4, D12, D20).

**Resolved (Phase C, C1 — `e0bbc4c`).** Architecture §6.1's write-path prose
now describes the post-B3 closure rule as the code implements it: a prior row
is closed only on a genuine contradiction — same object at opposite polarity
(any cardinality), or a functional predicate's different object at
both-positive polarity — and every other difference (multi-valued addition,
contrastive correction, both-negative) is a parallel write. §6.4 gained a
**Belief revision** subsection covering both the polarity path and the new
object-conflict path, including the functional-only constraint and the
negated-claim conservative-abstain asymmetry (Phase B Decision 1). §6.4 had
carried no belief-revision text at all before C1 — the subsection was added,
not edited. §6.1/§6.4 now agree with `tier_u.py` / `walker.py`. Detail in
`docs/phase_C_report.md`.

### D22 — Oracle audit logging is still gated on an unwired `audit_log` flag

Phase B's B3 finished the D8 cleanup for `tier_u.py`: the vestigial `audit_log`
constructor flag — which `build_pipeline` never set — was removed, and `TierU`
now logs via `log_event(db, …)` unconditionally (so `tier_u_row_closed`,
`tier_u_parallel_assertion` and `row_created` fire in the deployed pipeline).
The three substrate oracles — `predicate_translation.py`, `subsumption.py`,
`predicate_distribution.py` — still gate their `row_created` /
`row_generation_failed` events on the same unwired flag, so those events do not
fire in the deployed pipeline. v0.16 should apply the same cleanup to the
oracles.

**Resolved (Phase C, C2 — `65833f0`).** The vestigial `audit_log` flag was
removed from `predicate_translation.py`, `subsumption.py` and
`predicate_distribution.py`; the three oracles now log `row_created` /
`row_retracted` / `row_generation_failed` via `log_event(db, …)`
unconditionally, so those events fire in the deployed pipeline. C2's systematic
grep also found the dead `audit_log` constructor parameter — never read, gating
nothing — in seven further modules (`router`, `aggregator`, `kb_verifier`,
`kb_wikidata`, `python_verifier`, `resolver`, `walker`); per the C2 scope
check-in these were cleaned in the same cluster. No `audit_log` constructor
flag or `self._audit` attribute remains anywhere in `src/aedos/`. Three
deployed-pipeline tests (`tests/integration/test_oracle_audit_logging.py`)
guard the oracle events. No `log_event` event type needed adding — `log_event`
already accepts any event type. Detail in `docs/phase_C_report.md`.

### D23 — `lives_in` seed `single_valued` rationale invalidated by D16

`seeds/predicate_translation.json` classifies `lives_in` as `single_valued: 0`
with the rationale *"user_authoritative — single_valued is not consulted on the
user-authoritative route."* D16 makes that rationale false: the walker's
object-conflict path and the Tier U write path both consult `single_valued` on
the user-authoritative route. Phase B did not touch the seed pack — changing a
seed `single_valued` alters calibration inputs and is a deployment decision, not
walker/Tier-U code. v0.16 should revisit whether `lives_in` should be functional
now that the flag is consulted there, and correct the stale `reason` text either
way. Consequence to note: D16's headline `lives_in` example does not fire in the
*seeded* deployment until this is decided — but the *capability* is correct and
fires for every `single_valued = 1` predicate.

**Status (Phase C) — deferred to Phase 10.5 data.** D23 is held deferred, but
unlike the other deferred deltas it is not waiting on v0.16 design time — it is
waiting on empirical pressure from calibration. Whether `lives_in` should be
functional is a measurement question (how often an asserting party revises a
stated residence versus genuinely holds two), and the seed `single_valued`
value is a calibration *input*, so changing it before Phase 10.5 would alter
the very measurement that should justify the change. Phase C did not touch the
seed pack. D23 stays deferred until Phase 10.5 produces the data; the stale
`reason` text is corrected then, in light of it.

trusted by association. It was never audited and it was partial: two runners

The compounding defect: the **dry-run** (`pytest --run-calibration` without

**Mechanical fix — landed in the Phase D follow-up (`fixup-1`), not deferred.**

## From Phase E (open-weight migration comparison)

### D25 — DeepSeek V4 on OpenRouter is incompatible with disable-thinking + structured-output tool schemas

Discovered during the Phase E open-weight model comparison. Full diagnostic in
`docs/phase_E/deepseek_v4_flash_structural_errors.md`; summarised here as a
production-deployment constraint.

**The constraint.** DeepSeek V4 on OpenRouter (Flash and Pro variants, served
upstream by "Morph", a vllm-with-structural-tagging fork) cannot accept
`extra_body: {reasoning: {enabled: false}}` when the request also uses an
OpenAI-style structured-output tool schema containing variable-length (array)
fields. Morph's grammar compiler hits a C++ assertion
(`grammar_compiler.cc:1037`) — "*Invalid structural tag error: Only the last
element in a sequence can be unlimited, but the 1th element of sequence format
is unlimited*" — and returns a degenerate response with `choices: null` and a
502-coded error body. The harness sees this as a `TypeError` on
`resp.choices[0]`. Empirically: ~19% of calls hit it on
`extraction_corpus` (the harness's tool schema has a `claims` array, the
non-last-position-unlimited element the assertion names). Without the
`reasoning` payload, 0 of 11 retried cases hit it.

**The mechanism is mostly transparent.** When the request includes a
`reasoning` directive, Morph appears to construct the output grammar with a
reasoning section *prepended* to the tool's output schema; that puts an
unlimited-size reasoning block in non-last position when the tool itself also
has an array field. With no `reasoning` directive, Morph compiles a grammar
containing just the tool output, where the only unlimited element (`claims`)
*is* in the last position and the assertion holds.

**The Phase E post-fix run revised the picture.** Removing the `reasoning`
payload only halved the structural-error rate — it did not eliminate it.
The post-fix `deepseek-v4-flash × extraction_corpus` run produced **6/57 =
~10.5% structural errors**, split into two distinct failure modes:

- **(a) Morph grammar-compile errors persist without the payload.** 4 of the
  6 (~7%) hit the same "`Invalid structural tag … 1th element of sequence
  format is unlimited`" assertion. The bug is reachable from request shapes
  that don't include the reasoning directive — the payload was *a* trigger,
  not the only one.
- **(b) Empty-message responses.** 2 of the 6 (~3.5%) returned a parseable
  `choices: [{message: {tool_calls: null, content: null, reasoning: null}}]`
  — model gave up entirely. This signature didn't appear in the with-payload
  runs because most calls failed earlier at grammar compile.

**Final classification.** DeepSeek V4 models on OpenRouter (via Morph)
exhibit a ~10% structural-error rate even with reasoning disabled and the
payload removed. Two distinct failure modes: (a) Morph grammar compiler
rejects schemas with unlimited-length elements not in last position, fires
unpredictably even without the reasoning payload; (b) model occasionally
returns empty assistant messages with all fields null. **Not suitable for
Aedos's verification pipeline regardless of capability scores.**

**Phase E outcome.** Both `deepseek-v4-flash` and `deepseek-v4-pro` were
dropped from the comparison. `_CANDIDATES` carries an explicit `disabled`
field on each entry whose value is the disqualification reason;
`run_comparison` refuses to invoke a disabled candidate on a live run. The
comparison proceeded with four candidates (Devstral · Qwen · GLM-5.1 · Kimi).
The E4 synthesis should record that DeepSeek V4 was dropped on empirical
provider-infrastructure grounds, not on capability grounds.

**Implications for v0.16.**

1. **DeepSeek V4 on OpenRouter is unusable for soundness-critical purposes**
   (extractor, oracles, walker, python_verifier) at the observed ~10%
   structural-error baseline. Until the underlying issues are addressed
   upstream, v0.16 should not route any soundness-critical purpose to
   DeepSeek V4 on OpenRouter regardless of cost or capability advantages.
2. **Tool-schema redesign is a possible mitigation** (put the unlimited
   array field literally last, with no other variable-length elements
   anywhere in the schema). Reduces (a)'s frequency but cannot address (b).
   Would require restructuring `EXTRACTION_TOOL`, the predicate-translation
   tool, and the Python-verify tool, then empirical re-verification.
3. **An alternative endpoint** — DeepSeek's direct API, or a different
   OpenRouter-fronted provider for DeepSeek V4 if one comes online without
   the Morph layer — could side-step both failure modes. If v0.16 wants
   to reconsider DeepSeek, the path is verifying whether the issues are
   OpenRouter/Morph-specific or class-wide for the model family by testing
   a non-Morph endpoint.

**Status.** Both V4 variants `disabled` in `_CANDIDATES` with reasons
pointing to the Phase E diagnostic. v0.16 plan should treat this as a
production constraint on model selection — not a Phase-D-style runner
defect — and revisit only with evidence that the upstream issues have been
addressed or that a non-Morph route is available.

## From Phase F (deployment readiness audit)

### D26 — Proactive deployment-readiness audit as a standard pre-release pass

The v0.15 audit chain ran three distinct audit types:

1. **Verdict-production audits** (`audit_report.md`, `reaudit_report.md`,
   `reaudit2_report.md`) — does the pipeline produce sound verdicts?
2. **Fix-work audits** (`fixup_report.md`, `fixup2_report.md`,
   `fixup3_report.md`) — do the fixes do what they claim, without
   regressions?
3. **Measurement-instrument audits** (Phase D — `phase_D_report.md`,
   D24) — does the calibration runner score every case of its corpus?

Phase F adds a fourth: **deployment-readiness audits** — does the deployed
pipeline actually reach its external services, with the configuration each
service requires? The verification work was extensively audited; the
*wiring between the verification pipeline and the world outside it* was
not. Phase E surfaced the gap in passing: a `RUN_LIVE_KB=1` run produced
20 `NotImplementedError`s because `WikidataAdapter`'s three live methods
were stubs and `build_pipeline` constructed the adapter with no
configuration. Once Phase F started enumerating the integration surface,
further wiring-level gaps surfaced (`Config` never threaded through
`build_pipeline`, `AEDOS_KB_REQUEST_DELAY_MS` referenced by the runbook
but unread by any code, chat-wrapper end-to-end never exercised against
real services).

**The deeper pattern.** All three prior audit types were *reactive* —
they responded to gaps that surfaced when something real was run (the
build was complete, the audit's fixes were applied, the calibration
runner was exercised). Phase F is the first *proactive* audit: it walked
the integration surface before any deployment-readiness gap had a chance
to surface in mid-measurement and disrupt Phase 10.5.

The reactive pattern carries a particular cost: gaps surface inside the
measurement window, where every minute spent diagnosing "is this a system
bug or a deployment bug?" is a minute the measurement isn't producing
data. The live-KB stubs surfaced 20 cases into Phase E's derivation runs;
the calibration runners hard-`KeyError`'d on 74% of their corpora (Phase
D) and were caught only because Phase D was an audit pass, not a Phase
10.5 run. Both are cases where a *proactive* pass before measurement
would have surfaced the gap at a moment when the response cost was small
(fix the stub, fix the runner) rather than entangled with the measurement
the operator was trying to produce.

**v0.16 plan change.** The build process should include a
deployment-readiness audit pass as a standard gate before any live
measurement begins, paralleling D24's measurement-instrument audit. The
two are companion proactive passes:

- **D24's pass** — runner-vs-corpus key audit; every key the runner reads
  is present on every case of the relevant sub-category. Audits the
  measurement instrument.
- **D26's pass** — integration-surface inventory; for every external
  service the deployed pipeline depends on, the deployed pipeline
  reaches it, with the configuration that service requires, exercised
  by at least one live test. Audits the deployed system.

Together they convert end-to-end deployment readiness from a hope into a
checked invariant. v0.16's plan should specify both as pre-release gates
with named acceptance criteria, not as ad-hoc audit passes added after a
gap is discovered. The pattern Phase F establishes for D26 — three
questions per integration point (implemented? live-tested? matches
architecture?), with explicit `stubbed` / `mock-only` / `wiring-gap`
status categories — is a reasonable starting methodology for v0.16 to
inherit.

**Phase F outcome.** `docs/phase_F_plan.md` and
`docs/phase_F/deployment_readiness_audit.md` are the v0.15 artifacts;
they document the integration surface as of v0.15.0-rc.8 and serve as
the baseline for v0.16's pass.

### D27 — `_TEMPERATURE_DEPRECATED_PREFIXES` completeness

`llm/client.py:56` defines `_TEMPERATURE_DEPRECATED_PREFIXES =
("claude-opus-4-7",)`. Sonnet 4.6 / 4.5 and Haiku 4.5 also deprecated

`temperature=` is passed, which the deployed pipeline does not do. A
deployment that *does* call `rewrite(temperature=...)` against a Sonnet

will reject the request. v0.16 should either extend the list to cover
the deprecated-temperature Claude families or rework the check to a
positive list of temperature-accepting models. Found by F1.

### D28 — `Config.seed_file` consulted by deployed pipeline

`Config.seed_file` (`config.py:45`) is defined but no deployed-pipeline
component reads it. Seed loading is done by `seeds/load_seeds.py` as a
standalone script invoked from the runbook (Step 2). The
deployed-pipeline path (`build_pipeline`) does not consult

`build_pipeline` will get an unseeded pipeline. v0.16 should either
make `build_pipeline` honor `Config.seed_file` (loading seeds when

### D29 — Periodic consistency-check scheduler (architecture §5.4)

Architecture §5.4 specifies that consistency checks "run on a periodic
schedule (deployment-configurable; default once daily) and on-write."
The on-write path is wired via the `consistency_checker` parameter to
each oracle. `ConsistencyChecker.check_periodic()` exists at
`consistency.py:84` and scans every substrate table for conflicts —
but **no scheduler, background task, or deployment hook invokes it**.
Periodic re-scanning would catch conflicts that became stale after
subsequent writes (a row consistent at write time may conflict with a
row inserted later); without it, only the on-write window is checked.

Phase 10.5 does not depend on periodic checks running (the runbook
runs corpora sequentially within one session; on-write coverage
suffices for the measurement window). v0.16 should specify the
scheduling mechanism — likely a `croniter`-style entry, an apscheduler
task, or an external cron job calling a CLI entry — and the deployment-
configurable cadence. Found by F1 Mode 2.

### D33 (original analysis) — Fixture-encoded canonical entities unreachable via default live search

- **Fixture data:** "Obama" → search result 0 is Q76 (Barack Obama).
- **Live data (2026-05-20):** "Obama" top-10 is

  Q1414593, Q26446735, Q25095673, Q7074605]`. **Q76 (Barack Obama) is
  not in top-10.** Q41773 is "Obama, Fukui" (a Japanese town).
- **Fixture data:** "Williams College" → search result 0 is Q49112
  (Williams College, MA).
- **Live data (2026-05-20):** "Williams College" top-10 is
  `[Q49166, Q12072937, Q8021012, Q61782737, Q23018410, Q7989431,
  Q116718410, Q117012255, Q91769454, Q72585582]`. **Q49112 is not in
  top-10.**

**The core finding.** The fixtures encode canonical entities that
live Wikidata does *not* return in its default-ranked top-10 for the
same query. The deployed pipeline with `Config.wikidata_candidate_pool_size
= 10` cannot resolve "Williams College" → Q49112 today regardless of
downstream disambiguation logic — the entity is not in the candidate
pool at all. The issue is not "ranking diverged" (a milder framing
the first observation suggested); it is **"canonical entity absent
from candidate pool entirely."**

Multiple fixture-dependent tests and corpus cases may have been
silently masking this for the entire audit chain. The fixture surface
and the live surface have diverged beyond ranking; **the test suite
did not surface this until F2 added live tests, and the fixture-vs-live
divergence is structural, not just rank-positional**.

**Connection to Ambiguity A.** Phase F2 deferred type filtering at
entity resolution (Ambiguity A → defer to v0.16 pending Phase 10.5
disambiguation-error data). D33 is independent evidence — surfaced
*before* Phase 10.5 — that the disambiguation gap is real and
load-bearing. The Phase E `der_disambiguation_006` finding was the
first signal; D33 is the second. The deferral was right on cost
grounds; the *empirical pressure* to address in v0.16 is now stronger
than at deferral time.

**v0.16 work items.**

1. **Implement type filtering at the resolution boundary.** Architecture
   §9.3 specifies "type filtering driven by predicate metadata"; the
   v0.15 implementation defers this. v0.16 should land it. The
   implementation choice (post-filter via per-candidate P31 fetch vs.
   SPARQL pre-filter losing relevance ranking) was the trade-off
   recorded in Ambiguity A. D33's data argues the trade-off resolves
   toward implementing — even at the cost of additional HTTP calls
   per resolution.

2. **Audit `EntityResolver.select` LLM-disambiguation behavior.** The
   v0.15 mitigation is `EntityResolver.select`'s LLM-mediated selection
   when top candidates score within `_AMBIGUITY_GAP=0.15`. Verify
   empirically: when a canonical entity is present but ranks low
   (e.g. Q76 at rank 5 with score 1/6 = 0.167), do the top-2 scores
   fall within the ambiguity gap and trigger LLM selection? If not,
   the LLM mitigation never fires for the very cases it was designed
   to handle. The audit needs concrete cases from Phase 10.5's
   `entity_resolution_corpus` to evaluate.

3. **Periodic fixture-vs-live audit as D26-style proactive pass.**
   D26 established proactive deployment-readiness audits as a v0.16
   pre-release gate. D33 names a specific class of audit that gate
   should include: **fixture-vs-live consistency check**. For each
   significant fixture (entity search, statement lookup, subsumption
   chain), the pre-release pass runs the same query against live
   Wikidata and surfaces divergences — not as failures (live data is
   ground truth; the fixture is what's stale), but as visible signals
   that fixture-dependent test assumptions may need updating. This
   is the same pattern D24 named for runner-vs-corpus audits, applied
   to fixture-vs-source-of-truth.

**Phase 10.5 implication.** With Q76 and Q49112 unreachable in
top-10 for default queries, expect elevated false-abstain rates on
entity-resolution-heavy cases — particularly `entity_resolution_corpus`
and `kb_mapping_corpus`, both of which run under `RUN_LIVE_KB=1` per
the runbook. The LLM disambiguation layer (`EntityResolver.select`)
**cannot rescue cases where the canonical entity is absent from the
candidate pool** — it only chooses among returned candidates. Phase
10.5 calibration thresholds were set at deferral time without D33's
data; if they are not met, the gap traces to D33's pattern and the
right response is v0.16 work, not a v0.15 hotfix. Phase 10.5's report
should record entity-resolution false-abstain rates by predicate and
by canonical-entity-presence (was the expected Q-id in the pool at
all?) so v0.16's work can target the empirical distribution.

Concrete mitigations available at Phase 10.5 time *without* code
changes: raise `Config.wikidata_candidate_pool_size` from 10 to 50 (a
larger pool likely contains the canonical entity for popular queries),
or use longer/more-specific query strings in test cases. Both are
configuration-level workarounds, not architectural fixes — they delay
addressing the structural gap that v0.16's type filtering (work item
1 below) addresses properly.

**For F2's remaining implementations.** The pattern applies recursively.
F2 commit #2 (`_live_lookup`) should expect that live Wikidata may
return additional statements, fewer statements, or different qualifier
values than the fixtures encode. Tests for `_live_lookup` should
avoid over-specifying — no exact statement counts, no exact qualifier
values that aren't load-bearing for the verdict semantics, no
assumptions about preferred-vs-normal-rank distribution that the
fixture happens to encode. Verify the implementation reaches the
expected statements; do not test against fixture-frozen state.

Surfaced by F2 implementation commit #1.

### D34 — Audit fixture-authoring methodology

D33's finding raises a methodology question the audit chain never
addressed: **how were the v0.15 fixtures authored?** Two possible
explanations for the divergence between fixture-encoded canonical
entities and live-search results:

1. **Empirical drift.** Fixtures were authored from live API responses
   at some earlier time, and Wikidata's ranking has shifted since
   (new entities added, popularity signals changed, ranking algorithm
   updates). The fixtures captured a snapshot that has aged out.

2. **Aspirational authoring.** Fixtures were authored to encode the
   *right answer* the system should return — the canonical entity a
   well-functioning resolver would pick — rather than recording what
   the API actually returns at default `limit`. The test suite has
   then been measuring **system intent**, not **system behavior** —
   and the audit chain inherited this without questioning it.

These have very different implications:

- If (1), the issue is Wikidata drift and the v0.16 response is
  fixture refresh + periodic regeneration as part of the pre-release
  proactive-audit pass (D26 / D33 work item 3).
- If (2), the issue is deeper. The test suite verified that the system
  would behave correctly *if* fixture-encoded entities matched live
  data — but never verified the antecedent. v0.16's response includes
  fixture refresh *and* a discipline change: empirical-fixture
  authoring as standing convention. Every new fixture starts from a
  recorded live API response.

**v0.16 work.**

1. Determine which case applies for v0.15's fixtures. Examine
   `tests/fixtures/wikidata/README.md` and git history of the fixture
   files. The fixture format (`search_obama.json` has only 2 results,
   not the ~10 live wbsearchentities returns) suggests aspirational
   authoring — the fixture was hand-crafted to contain just the
   canonical entity plus a near-namesake.

2. If aspirational, regenerate fixtures from live API responses with
   the canonical-entity question resolved separately (type filtering
   per D33 work item 1; LLM disambiguation per D33 work item 2). The
   regenerated fixtures should preserve the *protocol contract*
   (response shape, qualifier semantics, rank handling) while
   recording *what the API actually returns*.

3. Establish empirical fixture-authoring as the discipline going
   forward. New fixtures start from a `curl` of the live endpoint at
   a recorded date; tests document the recording date in fixture
   metadata so future audits can detect drift.

**Connection to D26.** D33 captured fixture-vs-live audit as a v0.16
proactive-audit-pass item; D34 captures the upstream methodology
question of how fixtures should be authored to begin with. Together
they form the discipline: fixtures are recorded snapshots of live
state, refreshed periodically, with divergence checks as a standing
pre-release gate.

Surfaced by F2 implementation commit #1.

### D40 — Structural invariant: every Layer 4 source honors routing-hint authorization (F-042)

Phase F2's validation run surfaced **F-042**: the walker invoked the
Python verifier for every claim that didn't get a Tier U or KB
verdict, regardless of the claim's routing_hint. Architecture §6.5
step 3 specifies "Python verification *if the route is Python*"; the
walker dropped the gate. Combined with the live LLM-driven Python
verifier writing `return False` for subjective/preference/opinion
claims, this produced false `contradicted` verdicts — a §3.2
soundness violation. Five cases in the F2 validation hit this
pattern (`der_multihop_009`, `der_multihop_010`,
`der_predicate_translation_002`, `der_abstain_003`,
`der_abstain_006`).

The fix landed as a small commit (Phase F2 follow-up: F-042 routing
gate). Walker now consults `_predicate_routing()` and only invokes
the Python verifier when the routing hint is `python`. Four new
mocked tests (`TestF042RoutingGate` in `tests/unit/test_walker.py`)
guard against regression.

**Why the audit chain missed it.** Mock discipline. The test suite's
`MockTransport` Python verifier (`tests/conftest.py`) returns
canned responses appropriately for each test scenario; it never
exhibited the "LLM writes `return False` for subjective claims"
failure mode. The walker's unconditional Python invocation wasn't
visible because the mock was *too well-behaved* — it encoded
architecturally-correct verifier behavior. Two conditions had to
coincide for the bug to surface: walker doesn't gate (architectural
bug), AND live verifier doesn't abstain when LLM writes `return
False` for non-deterministic claims (live LLM behavior diverging
from mock discipline). This is the fixture-vs-live divergence
pattern from D33, applied to internal components.

**v0.16 work.**

1. **Structural test (the D26/D36 companion check).** Walk the
   walker code (and any other Layer 4 entry point) for `verify(`
   calls into sources; assert each is guarded by a routing-hint
   check. The check codifies architecture §6.5's per-route source
   authorization as a CI invariant. Ten-line `re` scan analogous to
   `test_purpose_table_completeness.py`.

2. **Architecture §6.5 wording.** Re-verify the architecture document
   names each source's authorization explicitly; the §6.5 wording is
   sometimes read as "fall through in order" rather than "gated by
   route." Cross-check against §4.2 (which is precise about routes).

Surfaced by F2 validation run. The five rescued cases now produce
`no_grounding_found` post-fix, matching architectural expectation.

### D41 — Adversarial mock fixtures for upstream-bug detection

D40 names the structural fix. D41 names the testing discipline that
would have caught F-042 *behaviorally* — independent of the
structural test.

The D40 structural test catches "walker code path doesn't gate
Python." But what if the gate exists but is buggy (e.g., always
returns True)? Or what if a future refactor introduces a parallel
invocation path? Structural tests catch named invariants; behavioral
tests catch consequences.

The behavioral test: for each load-bearing internal component, at
least one test fixture should simulate the failure modes the live
version could exhibit, **even if those failure modes aren't what a
well-disciplined implementation would produce**. For the Python
verifier, that means a `_AdversarialPythonVerifier` (landed in
`test_walker.py` as part of F-042's fix) that returns `contradicted`
when invoked — mimicking the live LLM's "`return False` on
subjective claims" pattern. Tests then assert the walker doesn't
propagate the adversarial output for routes that shouldn't have
invoked the verifier in the first place.

This catches F-042-class bugs from a different angle: even if a
future refactor reintroduces an unconditional Python invocation,
the adversarial fixture surfaces the consequence (false contradicted
on non-Python claims) rather than the structural defect (missing
gate). Together with D40, two independent tests guard the invariant.

**v0.16 work.** Establish adversarial mock fixtures as a discipline
for load-bearing internal components:

- For each Layer 4 source (Tier U, KB verifier, Python verifier),
  identify the failure modes the live version could exhibit that the
  current mock doesn't.
- For each, add an adversarial fixture that simulates that mode.
- For each, add a test that the walker / aggregator / consumer
  doesn't propagate the adversarial output when the architecture's
  routing or authorization rules would prevent it.

The principle: **mocks encode architecturally-correct behavior;
adversarial fixtures encode realistically-misbehaving behavior**.
Both are needed for tests to catch upstream bugs that depend on
mock vs live divergence.

Surfaced by F-042's investigation. The fixture pattern landed in
`tests/unit/test_walker.py` (`_AdversarialPythonVerifier` + four
`TestF042RoutingGate` tests) is the v0.15 worked example for v0.16
to generalize.

**F3 refinement (test-pollution-via-inheritance class).** F3 commit
#4 surfaced a sibling pattern: `_ComparisonHarness(_Harness)` inherits
`_Harness.__init__()`; when F3 added an `.env`-load call to that init,
unit tests of the Phase E comparison harness (`pec.run_comparison(
load_env=False, transport=...)`) silently triggered the load through
inheritance, polluting `RUN_LIVE_KB=1` into process env and breaking
downstream fixture-mode tests. The fix was conservative (revert the
auto-load); the structural-test invariant this would have caught:
"shared base-class `__init__` does no environment / filesystem
side-effects that affect downstream tests." A grep-walk for
side-effecting init methods on base classes, with explicit opt-out
markers for cases that legitimately need them, would fit alongside
the D40 gate. v0.16 candidate alongside D41's adversarial mock work.

### D43 — LLMClient tool-call compliance robustness across OpenRouter providers

`LLMClient.extract_with_tool` (OpenAI-compatible path) raises
`RuntimeError("extract_with_tool: no tool call in OpenAI-compatible
response")` when `message.tool_calls` is empty. Phase E3's Kimi K2.6
extraction run exhibited this at 42% (24/57) — the model emitted the
correct `{"claims": [...]}` JSON as `message.content` instead of
calling the tool. Diagnostic found the failures were **100% routed
through OpenRouter's WandB provider** (the other 33 succeeded; WandB
also serves Kimi K2.6 but the canonical run happened to route a slice
through it).

The v0.15 mitigation was per-candidate: `_CANDIDATES["kimi-k2.6"]` now
includes `extra_body: {"provider": {"ignore": ["WandB"]}}`. This is
sufficient for the Phase E comparison and the chosen Phase 10.5
configuration — but it is a candidate-config patch, not a client-layer
fix. The longevity story for open-weight model migration needs a more
durable answer.

**The class of failure.** OpenRouter routes each request to one of N
provider deployments per model (Kimi K2.6 has 18). Each deployment is
a separate vLLM/sgLang/proprietary stack with its own tool-call
implementation. Providers vary in compliance: some always emit
`tool_calls`, some always emit `content` carrying the same JSON, some
mix. The client's strict-tool-call contract presumes uniform compliance
across providers — empirically false.

**v0.16 work.** Three complementary mitigations at the client layer:

1. **Content-fallback parse with schema validation.** When
   `message.tool_calls` is empty and `message.content` parses as JSON
   matching the tool's `parameters` schema, accept the parse; emit a
   `tool_call_fallback_used` audit event capturing the model, provider
   (when OpenRouter exposes it), and tool name. The schema validation
   guards against accepting arbitrary text as a tool result — a strict
   `jsonschema.validate` against the registered tool schema, not a
   key-presence check. Tests: tool_calls-present (golden path),
   content-only-valid-schema (fallback path), content-only-malformed
   (raise), content-only-schema-mismatch (raise), both-empty (raise).

2. **Provider observability in production.** Capture `raw_response.
   provider` per LLM call into the `CallRecord` (or alongside it) and
   surface a provider × tool-call-mode breakdown in the metrics layer.
   Phase E's F3 harness fix did this diagnostically — generalising to
   production lets a WandB-class issue surface from telemetry rather
   than from a 42%-rate Phase E run.

3. **Per-purpose default provider exclusions.** `DEFAULT_MODEL_BY_PURPOSE`
   gains an `extra_body` slot whose `provider.ignore` accumulates the
   list of providers Aedos has empirically observed dropping tool-call
   compliance. The Kimi/WandB pair is the seed entry; the list grows
   as observability (item 2) surfaces new pairs.

The three compose: observability (2) surfaces the next provider that
needs exclusion; the per-purpose ignore list (3) is the place to record
it durably; the content-fallback (1) is the safety net for the calls
that route to a bad provider before the exclusion catches up.

Surfaced by Phase E3's Kimi K2.6 diagnostic. Documented in
`docs/phase_E/kimi_k2_6_wandb_provider_diagnostic.md`.

### D44 — Extractor capability gaps surfaced by Phase E extraction triage

Phase E3's per-case triage of Haiku 4.5 × extraction_corpus surfaced
two categories of "missed case" that are NOT model failures but
**capabilities the v0.15 extractor doesn't have by design**. Two
corpus cases were removed in the Phase E corpus cleanup pass
(2026-05-23) because they require behavior outside the extractor's
contract; both are restated here as v0.16 candidate work.

**Gap 1 — Meta-attribution for document-context first-person.**

Original case (`firstperson_002`, removed): "We found in our study
that aspirin reduces fever" with `asserting_party="doc:study_001"`,
`expected_subject="doc:study_001"`. The corpus expected the extractor
to emit a meta-claim where the document itself is the subject (the
document asserting that aspirin reduces fever), rather than the
inner factual claim (aspirin → reduces → fever).

The v0.15 extractor's contract is "extract verifiable factual claims
from text"; the `asserting_party` is provided externally by the
caller (chat / document / deployment context) and threaded through
`ExtractionContext`. The extractor does not synthesize "X
asserted Y" claims by treating the first-person plural in the text
as a reference to the asserting party. That would be a different
kind of claim (an assertion-about-an-assertion) requiring
architectural support v0.15 doesn't have.

**v0.16 work.** Decide whether meta-attribution claims are
in-scope for the extractor or for a separate layer:
- If extractor: add a "meta_attribution" rule + claim shape, where
  document-context first-person ("we", "our", "I" in a doc) emits
  both the inner claim AND a meta-claim with the doc as subject.
- If separate layer: route document inputs through an attribution
  pass that wraps the extracted inner claims with the doc as
  asserter, leaving the extractor alone.

The choice affects how Layer 1 composes with document ingestion
(v0.16 candidate). The Phase 10.5 corpus should reintroduce a
test case once the contract is decided.

**Gap 2 — Active-to-passive voice pivot for first-person object.**

Original case (`firstperson_009`, removed): "The company hired me"
with `asserting_party="user_test"`, `expected_subject="user_test"`.
The corpus expected `user_test` to appear as a subject in some
claim. The v0.15 extractor's literal extraction is `the company →
hired → me`, where "me" is the object. The expected behavior
requires either:
- A passive-voice rewrite ("user_test was hired by the company"),
  or
- Treating "me" as a subject even when it's grammatically the
  object (a contextual rewrite into "user_test was-hired-by the
  company").

Neither is currently in the extractor's contract — both would
violate the "verbatim source_text" discipline that grounds
extracted claims in the input text. The extractor's first-person
canonicalization at `extractor.py:_canonicalize` only fires on
subjects; this is deliberate because synthesizing claims with the
asserting party as subject when the grammar makes them an object
risks inventing facts.

**v0.16 work.** Decide whether the extractor should pivot voice
for first-person objects, or whether downstream consumers should
handle "me as object" semantically:
- If extractor pivots: add a passivization rule with the new
  `passive` claim shape. Increases the extractor's responsibility.
- If consumer handles: document that "me in object" produces a
  literal claim with the company as subject, and downstream code
  that needs "things about user_test" must include object-position
  searches.

Related to gap 1 — both are about "who counts as a subject" in
first-person contexts. The cleanest design likely treats them
together.

**Related — `firstperson_008` and the multi-participant first-person
gap (removed in cleanup; v0.16 work item).**

A third case ("me and Alice went to MIT" → `expected_subject="user_test"`)
was kept through the v3 prompt iteration but removed when the v4 prompt
re-surfaced the underlying mismatch. Three reasons made it a poor
measurement instrument:

1. **Linguistic ambiguity.** "me and Alice went to MIT" admits two equally
   defensible extractions: parallel-independent ("me went to MIT" + "Alice
   went to MIT") or joint event (participants=[I, Alice]). Modern LLMs
   tend toward the joint reading; the corpus contract assumed parallel.
   v2 happened to emit parallel; v3 emitted joint and failed. The
   pass/fail is determined by which linguistic reading the LLM happens
   to choose, not by model capability.

2. **Two concerns conflated.** The case combines "canonicalize 'me' (a
   non-nominative first-person form)" with "handle coordinated subjects
   with first-person", and the second concern dominates which extraction
   shape the LLM produces.

3. **System gap under the joint reading.** When the LLM emits the joint
   shape (subject="I", participants=["I","Alice"]), `decompose_event`
   discards the subject and replaces it with an event_id; "I" survives
   only as the OBJECT of a `has_participant` claim. `_canonicalize`
   fires on subjects only, so the asserting_party never enters the
   claims as a subject and the runner check `any(c.subject == "user_test")`
   fails.

The v0.16 work this carves out: decide whether the extractor should
canonicalize first-person tokens through the participant-decomposition
pipeline (e.g., have `decompose_event` substitute the asserting_party
for first-person tokens in `participants` before emitting `has_participant`
claims), or whether the corpus and runner should accept "user_test as
participant" as equivalent to "user_test as subject" for first-person
test cases. Both involve coordinated changes to the extractor design
and the corpus contract; both are out of scope for v0.15.

Surfaced by Phase E3's per-case extraction triage on Haiku 4.5,
2026-05-23. The corpus cleanup passes that removed `firstperson_002`,
`firstperson_009`, `firstperson_008`, and `decomp_008` are recorded in
each case's removal commit; the `decomp_006` input was modified to
include named participants (also documented in the corpus's `notes` field).

### D45 — Component-prompt discipline: explicit non-triggering conditions

Phase E3's extraction-prompt iteration (baseline → v1 → v2 → v3 → v4 → v5)
exposed a pattern that should govern every LLM-driven component prompt
in Aedos: **rules encoding Aedos-specific contracts must specify both
positive triggering conditions AND explicit non-triggering conditions**.
Without explicit non-triggering conditions, rules over-apply at adjacent
linguistic patterns and the prompt iteration trades one regression for
another instead of monotonically improving.

**The empirical pattern** (documented in `docs/phase_E_report.md` and
the per-iteration result directories `docs/phase_E/results/
augmented_prompt_v{1..5}/`):

| Iteration | Lift | New regression | Cause |
|---|---:|---|---|
| baseline → v1 | +5 (temporal) +1 (decomp) | -2 (norm_014, fp_005) | reify rule too broad |
| v1 → v2 | +3 (fp_005, fp_008, norm_014) | -1 (decomp_002) | reify-or-not signal under-specified for "and" coordinations |
| v2 → v3 | +3 (decomp_002/006/007) | -2 (decomp_010, fp_008) | uniform-reify ignored multi-emission failure mode |
| v3 → v4 | +1 (decomp_010) | -2 (decomp_003, norm_005) | anti-example for multi-emission caused over-reify of binary relations + verb-stem extraction |
| v4 → v5 | +2 (decomp_003, norm_005) | 0 | binary-relation guard + location-vs-year clarification (closed both regressions) |

Every iteration v1-v4 closed the targeted case AND opened a new failure
mode at an adjacent linguistic pattern. v5's success came from rules that
explicitly addressed both directions: "REIFY when ≥2 named participants
performing an action" AND "DO NOT REIFY binary relations / single-
participant cases / copular constructions." The "do this" half alone
was insufficient.

**Concrete rule shapes that produce stable lifts:**

1. **Positive trigger + explicit non-trigger.** "For X, do Y. Do NOT
   do Y for Z (X-adjacent case)." Example from v5 rule 10: reify
   multi-participant events (positive) but explicitly not binary
   relations (negative).
2. **Side-by-side CORRECT/WRONG examples for structural decisions.**
   Side-by-side anti-examples close model-interpretation gaps that
   prose alone misses. Example from v5 rule 10: explicit "CORRECT —
   ONE extraction" and "WRONG — three extractions" labeled examples
   for the multi-emission failure mode.
3. **Discriminate on examples that differ minimally.** "Apple was
   founded in 1976 → predicate='was founded', valid_from='1976'"
   adjacent to "Apple was founded in Cupertino → predicate='was
   founded in', object='Cupertino'" (hypothetical). Same verb,
   different object class, different rule outcome. Model needs to
   see the discriminator explicitly.
4. **Co-locate the discriminator with the rule that needs it.** Rule 7
   (temporal field usage) was the cause of norm_005's regression at
   v4 because the rule's example focused on the year case and was
   silent on locations. v5 added the location example inside rule 7
   itself, not in a separate rule.

**v0.16 work item — audit all LLM-driven component prompts against
this discipline.**

The components that pass system prompts to LLMs in v0.15 (audited
2026-05-23):

- `src/aedos/layer1_extraction/extractor.py` — `_SYSTEM_PROMPT`
  (extraction). **Phase E5 (this commit) raises this to the v5
  discipline. 53/53 on cleaned corpus.**
- `src/aedos/substrate/predicate_translation.py` — predicate
  metadata generation prompts. **Not yet audited; v0.16 candidate.**
  Phase E numbers (51.3% on `predicate_metadata_corpus` with GLM,
  48.8% with Kimi) suggest similar under-specification gaps. The
  "1 of 39 failures was `pred_kb` (KB-route translation)" pattern
  in Phase E's GLM data is reminiscent of extraction's
  "rule-vs-corpus mismatch" pattern. Worth auditing.
- `src/aedos/layer4_sources/python_verifier.py` — Python verifier
  prompt. **Not yet audited.** Devstral's 90% (0 false-verifieds)

  `python_verification_corpus` suggests the prompt is already
  near-ceiling for one model class but not others. Prompt audit
  might lift the lower performers.
- `src/aedos/layer4_sources/walker.py` and substrate oracles —

The audit pattern is the same one Phase E3 used for extraction:

The expected return on investment is substantial — extraction's

no architectural changes and modest API spend (~$0.78 across 5

### D47 (original analysis) — Contextual disambiguation of bare ambiguous strings upstream of KB queries

**Surfaced by Phase G D33 live validation, 2026-05-23.** D33's
implementation surfaced that some canonical entities (Q76 for Barack
Obama, Q49112 for Williams College) are not reachable from their bare
ambiguous string forms via Wikidata. The bare strings are not registered
in Wikidata's label or altLabel fields for the canonical entities, by
design — verified by direct SPARQL on 2026-05-23, e.g. Q76's only
English altLabel is "POTUS 44".

This means resolution of bare ambiguous strings must happen upstream of
KB queries — at extraction time or in the entity resolution oracle with
surrounding-claim context. The D33 type filter and SPARQL fallback are
necessary but not sufficient: when the canonical entity's label set in
Wikidata doesn't include the asserted reference, no amount of post-hoc
filtering rescues it.

**Empirical pin tests.** Two D43-pinning xfail tests in
`tests/integration/live/test_wikidata_live.py::TestD33TypeFilterAndBareStringLimit`:

- `test_obama_short_query_does_not_yield_canonical_q76` — pins that
  bare "Obama" + type filter [Q5] does not reach Q76.
- `test_williams_college_short_query_does_not_yield_canonical_q49112`
  — pins that bare "Williams College" + type filter does not reach
  Q49112.

Both `strict=False` so an xpass (Wikidata edits adding the missing
altLabels, or a Phase 10.5 disambiguation improvement) reports as a
visible notice.

**v0.16 work items.**

1. **Extraction-time normalization.** When an extracted subject
   reference is short and ambiguous, expand it using the surrounding
   source text and asserting-party context (e.g., "Obama said X" in a
   2010 political-news article → "Barack Obama"). This is an extractor
   responsibility, not a resolver responsibility — the resolver only
   sees a single reference string today and cannot recover lost context.

2. **Oracle context enhancement.** The entity-resolution oracle
   (`EntityResolver.select`'s LLM call path) currently receives only
   the reference and candidate list. Extending it to also receive the
   asserting party, the source text snippet, and prior-resolution
   context would let it choose between same-name candidates (e.g.,
   three different "Obama"-named humans surfaced by the type filter).

3. **Abstention fallback for unreachable canonical entities.** When the
   type-filtered pool returns multiple low-confidence same-type
   candidates and no high-confidence pick exists, the resolver should
   abstain honestly rather than confidently select the wrong Q-id. The
   current threshold `_SELECT_THRESHOLD=0.6` and `_AMBIGUITY_GAP=0.15`
   were calibrated for the fixture-encoded canonical-present case; the
   D43 finding suggests they need re-evaluation with the
   filter-returns-wrong-candidates pattern in mind.

**Phase 10.5 implication.** Phase 10.5 corpora typically use
unambiguous references in the canonical case; for those, the D33 type
filter is purely additive (no regression on the working path). Corpora
that include bare ambiguous references will produce abstentions on
those specific cases until D47 lands. Interpret as honest measurement
of v0.15's known constraints, not as a v0.15 defect. The
`docs/phase_10_5_runbook.md` carries the operator-facing version of
this note.

### D48 — Audit each calibration runner's pipeline-traversal shape vs. the corpus's intent

**Surfaced by Phase H D47 planning, 2026-05-23.** The pattern, stated
sharply: **corpora should exercise components in the mode the
components are actually used in by the deployed pipeline.** Mismatches
like `entity_resolution_corpus` calling `resolver.resolve()` with bare
references (which the deployed pipeline never does post-D47, because
extraction always produces references with source-text context that
flows into the resolver) are *measurement artifacts that the audit
chain did not catch*. The corpus number measures the runner's
synthetic shape, not the deployed system's behavior on equivalent
input.

**How this surfaced in D47.** D47 needed a wiring decision: should the
Wikipedia normalizer fire only between `Extractor.extract` and the
substrate (pipeline-only), or also inside `EntityResolver.resolve`
(every caller benefits)? The operator chose the latter because
`_run_entity_resolution` and other calibration runners invoke
substrate components directly with bare references, bypassing
extraction. A strict pipeline-only D47 would have left
`entity_resolution_corpus` measuring exactly the same accuracy
post-D47 because the runner doesn't traverse the new path.

The resolver-internal wiring is correct for the deployed pipeline AND
the corpus runner — they both call `resolver.resolve` — but it
papered over the underlying mismatch: the corpus shape doesn't match
the deployed pipeline's shape for equivalent input. The deployed
pipeline never asks the resolver "what is 'Obama' (with no other
information)?" — it asks "what is 'Obama' (here is the source
sentence, here is the predicate context, here is the asserting
party)?". The corpus runner asks the former. Post-D47, the resolver
has the *capacity* to use source-text context (via `LocalContext`),
but the corpus runner provides None, so the disambiguation power D47
adds is invisible to the measurement.

This is the second instance of D24's class (runner-vs-corpus-key
mismatch) in the v0.15 audit chain: pipeline-step features (D33 type
filter, D47 normalizer, future pipeline steps) only manifest when the
runner exercises the layer that hosts them, **and provides the
upstream inputs the deployed pipeline would have provided**. A runner
that bypasses an upstream layer — or that calls a substrate component
directly without populating the request-scoped context the deployed
pipeline would populate — silently misses the feature regardless of
whether the deployed pipeline would have used it.

**v0.16 work item.** Audit each calibration runner against:

1. **What pipeline shape does the corpus's input + expected_output
   imply?** A corpus case whose input is a bare reference + predicate
   implies "the deployed resolver answer for this reference"; a corpus
   case whose input is a sentence implies "the deployed pipeline's
   verdict for this sentence." The runner must traverse the same
   shape.
2. **Does the runner invoke each layer the deployed pipeline would
   touch for that input?** A runner that calls
   `resolver.select(resolver.resolve(...))` directly skips extraction,
   the walker, the verifier, the aggregator — all of which the
   deployed pipeline runs for the same input.
3. **For corpora whose runner shape genuinely should be
   substrate-direct** (the corpus pins a substrate-component property,
   not a system-level verdict), ensure all upstream-of-substrate
   features that the runner needs to opt into are wired through the
   `LocalContext` (or equivalent component-input type) the runner
   passes. D47's `source_text`, D33's `expected_entity_types`, and
   future similar inputs all fall into this category. **And** ensure
   the corpus carries the data those fields need; an
   `entity_resolution_corpus` case that pins "Obama → Q76" with no
   surrounding text is genuinely under-specified for a system that
   uses surrounding text to disambiguate.
4. **Decision per corpus**: either (a) lift the corpus to the
   pipeline shape (add source text to bare-reference cases; have the
   runner traverse the full pipeline), or (b) keep the corpus
   substrate-direct and document the divergence explicitly as
   "measures component capacity, not deployed-system behavior." Both
   are acceptable; the audit's job is to make the choice visible
   rather than letting it default silently.

The audit should produce a per-corpus traversal-shape document, with
gaps flagged as upstream-shape findings. Phase 10.5 (or the v0.16
build that lands D48) consumes the document to add the missing
plumbing or accept the runner-vs-pipeline divergence with an explicit
note in the validation report.

**Concrete v0.15 instance to lift in v0.16.** The
`entity_resolution_corpus` post-D47 result (82%) is bounded by exactly
this mismatch: the 9 failing cases ALL fall into either Wikidata-side
limits or the corpus's bare-reference-no-context shape that bypasses
D47's Stage 2. Adding source-text fields to the bare-reference cases
(and updating the runner to thread them through `LocalContext`) would
let the corpus measure what the deployed pipeline actually does for
equivalent input.

**Soft observation**: this is the third proactive-audit class to
emerge in v0.15 (after D24's runner-vs-corpus-keys and D26's
deployment-readiness). The pattern is consistent: every audit done
inside the v0.15 chain has surfaced at least one class of audit
v0.15 was missing. v0.16's pre-release gates should institutionalize
D24 + D26 + D48 as a standing trio.

### D49 — Calibration corpora exhibit run-to-run variance; report ranges, not points

**Surfaced by Phase H D47 step 5 (2026-05-23).** The official
`derivation_corpus` calibration run reported 19/50 = 38%. A separate
diagnostic run with identical configuration produced 17/50 = 34% — a
4-percentage-point spread between two runs of the same code on the
same corpus with the same environment.

The variance traces to **non-determinism in the multi-LLM extraction
+ walker chain**. The derivation runner chains Haiku 4.5 (extraction)
→ Qwen3-Next via OpenRouter (substrate oracles) → Devstral via
OpenRouter (Python verifier), plus per-case Wikidata calls. Small
differences in any LLM's tool-call output flow into different walker
traversals, different KB queries, different verdicts. The chain is
inherently stochastic; a single calibration run is a *point estimate*
on a noisy underlying distribution.

This is not a new finding in spirit — the medium-bar evaluation
already mandates ×3 median runs (recorded in the Phase 10.5 runbook
under medium-bar protocol) for exactly this reason. What is new is
**evidence that the per-corpus calibration runs also exhibit
non-trivial variance** and need the same discipline. Until this
session, calibration accuracies were reported as point values
("entity_resolution: 82%", "derivation: 36%"); the implicit
assumption was that the corpus runner is deterministic enough that
one run = the truth. D47's measurement disproved that for
derivation_corpus.

**v0.16 work item.** Promote ×3-median (or larger, depending on
corpus variance) reporting from the medium-bar evaluation to the
per-corpus calibration runner, for any corpus whose verdict path
involves more than one LLM call. Specifically:

1. **Identify variance-prone corpora.** Any corpus whose runner makes
   more than one LLM call per case is a candidate: at minimum
   `derivation_corpus` (extraction + walker + KB + python). Likely
   also `extraction_corpus` (extraction prompt non-determinism),
   `consistency_check_corpus` if it routes through oracles, and any
   future corpus that chains components.
2. **Add ×3-median reporting** to the calibration runner, with a
   per-corpus knob defaulting to 1 (preserving Phase 10.5's current
   single-run behavior for corpora that don't need it). Update
   `test_corpus_runner.py`'s `test_corpus_calibration` to invoke the
   runner N times and report median, min, and max.
3. **Update the Phase 10.5 runbook** to record the range, not the
   point, in the threshold table. A corpus whose median is below
   threshold but whose max is above is a different signal than a
   corpus whose median is at threshold and whose min is far below.
4. **Update D17's `THRESHOLDS` table contract** to accept either a
   scalar (single-run, deterministic-enough) or a (median, tolerance)
   tuple. The doc-test in `tests/unit/test_runbook_thresholds.py`
   updates correspondingly.

**Phase 10.5 implication.** Without ×N reporting, two consecutive
Phase 10.5 runs can produce headline numbers that differ by several
percentage points without anything having changed in the code. Phase
10.5's narrative ("D47 lifted entity_resolution by X%") needs the
variance band to be honest, and the medium-bar discipline gives the
right template.

**Concrete v0.15 instance.** D47's Phase H validation document
(`docs/phase_H/d47_validation.md`) records the derivation_corpus
result as "34-38% post-D47" rather than a point — this is the right
shape for variance-prone results, and v0.16's runner upgrade should
make that shape standard.

### D50 — Audit every corpus runner for cross-case state isolation

**Surfaced by Phase H D16 (2026-05-23).** The Phase H D16 diagnostic
discovered that `_run_derivation` in `tests/calibration/test_corpus_runner.py`
was leaking Tier U rows and case-seeded subsumption rows across the 50
derivation cases via the harness's shared in-memory DB. The fix

artifact (2 universal false-verifieds on `der_revision_001`/`_002`).
The general pattern — case-state vs substrate-state separation —
deserves a systematic audit across **every** corpus runner.

**Pattern statement.** Each `_run_<corpus>` function in
`test_corpus_runner.py` (and any harness inheriting `_Harness` via the
shared `_RUNNERS` dispatch) must be classified by what it mutates:

- **Substrate-state mutations** (intentional cold-start measurement

  `entity_resolution_cache`, LLM-generated `subsumption` rows

  Phase 10.5 runbook explicitly notes "calibration measures the LLM's
  cold-start substrate-row generation, so seeded predicate
  translations would mask the signal."
- **Case-state mutations** (must be isolated per case): `tier_u`,
  case-seeded `subsumption` rows (`source='calib'`), and any future
  per-case state a new corpus might introduce. These should reset
  at the start of each case to match the case's setup intent.

The Phase H D16 fix addressed `_run_derivation`. The remaining
runners need the same lens:

- `_run_extraction` — appears stateless; uses Extractor + ExtractionContext,

- `_run_predicate_metadata` — calls `consult()` only. Substrate-state

- `_run_temporal_scope` — same shape as `_run_extraction`. Confirm.

- `_run_kb_mapping` — `consult()` only. Substrate. Confirm.
- `_run_subsumption` — calls substrate.subsumption.consult. Mixed:

- `_run_predicate_distribution` — `consult()` only. Substrate.

- `_run_python_verification` — calls PythonVerifier. Stateless.

- `_run_derivation` — fixed in Phase H D16.

### D51 — Reverse KB neighbor enumeration

**Surfaced by Phase H D5 validation (2026-05-24).** D5 implemented
KB neighbor enumeration as the walker's fourth KB operation, with
outgoing-edge SPARQL: `SELECT ?value WHERE { wd:E ?prop ?value . }`.
The full-corpus diagnostic
(`docs/phase_H/d5_diagnostic.json`) revealed that **D5 never fires
on the `part_of` relation in `derivation_corpus`**, because the
locative predicates' distribution verdict is `distributes_up`
(directions={"child"}) and D5's outgoing-edge enumeration produces
*parents* of E (the wrong direction). The corpus's `der_multihop_*`
cases — the architectural target for D5 — all use locative
predicates and would benefit from CHILD-direction enumeration:
`SELECT ?value WHERE { ?value wdt:Pn wd:E . }`.

**v0.16 work.** Add a `enumerate_neighbors_reverse(entity,
properties)` method (or extend `enumerate_neighbors` with a
`direction` parameter — design choice for v0.16) returning the list
of entities that point TO `entity` via each property. Walker
integration: in `_expand_via_kb_neighbors`, also fire when
`"child" ∈ directions`, calling the reverse enumeration.

**Defense-in-depth needed.** Reverse enumeration can be unbounded
(P17=Q30 has millions of entities). The implementation needs a
LIMIT clause and ideally a property-specific cap (e.g. P361/P131 are
fanned narrowly; P17 is fanned widely and should have a tighter
cap or be excluded from reverse enumeration entirely).

**Phase 10.5 implication.** If Phase 10.5's `derivation_corpus`
×3-median measurement shows `der_multihop_*` cases consistently
abstaining for the documented reason, D51 is the highest-leverage
v0.16 work item to address that pattern.

### D53 — Replace Wikipedia disambig-page candidates with Wikidata `wbsearchentities`

**Surfaced by Phase H Cluster 1 (2026-05-24).** The Cluster 1 diagnostic
found the dominant entity-resolution failure mode was Mechanism C
(candidate list missing the canonical entity), not Mechanism A
(over-calibrated Stage 2 abstention). D47 Stage 2 was doing the right
thing — when it abstained, the right answer genuinely wasn't in the
candidate set. Cluster 1's intervention raised the candidate cap
20→100 and added an implicit-disambig probe for canonical_no_redirect
outcomes (e.g. "Apple" → also probe "Apple (disambiguation)"). Those
fixes are working at the Stage 2 layer but don't address the
underlying source problem: **Wikipedia disambiguation pages are
human-readable artifacts whose link lists were never designed as
programmatic disambiguation APIs**. Truncation, alphabetical ordering,
and inclusion biases produce systematic misses. The cluster 1 fix
papered over the symptom; D53 is the architectural fix.

**Replacement.** Wikidata's `wbsearchentities` is the architecturally
appropriate candidate source: ranked by entity prominence, returns
type metadata usable for predicate-driven filtering, designed for
disambiguation as a programmatic API. The shape would be roughly:

1. Stage 1 (Wikipedia redirect resolution) remains as a fast-path for
   unambiguous bare references (`Obama` → `Barack Obama`, `USA` →
   `United States`).
2. On `disambiguation_page` (Wikipedia returns the disambig page) or
   on the implicit-disambig probe firing — instead of scraping the
   disambig page's link list, call `wbsearchentities?search={surface}`
   and feed the ranked candidates (with their Wikidata labels +
   descriptions + type Q-ids) to Stage 2.
3. Stage 2's prompt becomes richer: each candidate carries a short
   description (Wikidata's snippet) and its P31 type Q-ids that Stage
   2 can use to filter against the predicate's expected_entity_types
   (the D33 type filter signal).

**Scope.** Effectively D47 v2. The current `_fetch_disambiguation_candidates`
is replaced; `_stage_2_llm_select` builds a different user_message;
audit log shape gets a new field (`candidate_descriptions`). The
Stage 2 abstain/select semantics stay the same. New live tests for
the wbsearchentities path; existing tests stay green via the Stage 1
fast path.

**Priority.** Defer to v0.16; Phase 10.5 entity_resolution measurements
will inform how much residual gap remains after Cluster 1's bandaids
(higher cap + implicit-disambig probe). The cap-bump may already
solve most truncation cases; D53 is the architecturally correct
replacement when (a) cases still fail post-Cluster 1, or (b) the
operator wants to retire the disambig-page-scraping codepath.


### D54 — Tertiary measurement: hybrid-vocabulary mixed deployment

**Surfaced by Phase H Cluster 3 (2026-05-26).** Cluster 3 established
the dual-measurement framework: derivation_corpus (and the other
calibration corpora when applicable) runs in two modes — seeded

The framework's natural third measurement, deferred from v0.15: a

- Report a third measurement curve as a function of seed-pack

**Scope.** New runner parameter (`seeded_fraction: float | list[str]`)

`scripts/cluster_3_validation.py`) to sweep the curve; documentation
of the hybrid number alongside seeded and cold-start in the Phase
11 (post-rc.11) release narrative.

**Priority.** Defer to v0.16. v0.15's release decision works on the
two-point measurement (seeded + cold-start); the hybrid curve adds
nuance but doesn't change the ship decision.

### D57 — Functional-at-a-point-in-time cardinality

**Surfaced by Phase H Cluster 3 diagnostic (2026-05-26).** The
diagnostic for `der_revision_002` ("Asa works at Google" with prior
"employed_by Microsoft", expected contradicted) found the system
treats `employed_by` as multi-valued (`single_valued=0`) because a
career history may have many employers, while the corpus author
encoded a point-in-time functional semantic (one current employer;
relocations / employment changes are belief-revision events). Both
cardinalities are legitimate for the same predicate at different
abstraction levels.

**v0.16 plan change.** Introduce a third predicate cardinality
between `functional` and `multi-valued`:
**functional-at-a-point-in-time**. Semantics:

- A subject has AT MOST ONE object value at any given time interval.
- Changes over time are valid (the relation can have different
  values during disjoint time intervals).
- A new claim with no scope override that contradicts the current
  positive object value fires belief-revision (the prior closes;
  the new claim opens). This is what Cluster 2's object_conflict
  path already does for `single_valued=1`, but with explicit
  point-in-time semantics it also handles employment, party
  affiliation, occupation, religion (currently all multi-valued in
  the seed pack but with the same "current" semantic).

**Affected seed entries.** Cluster 3 lists `employed_by`,
`member_of`, `political_party`, `occupation`, `religion`,
`headquarters_in`, `nationality` as candidates for reclassification.
The v0.16 audit re-examines each to decide whether the corpus's
expected semantic matches functional-at-a-point-in-time vs. genuine
multi-valued.

**Schema change.** `predicate_translation.single_valued` becomes a
3-value column (or a new column `cardinality` with values
`functional` / `temporal_functional` / `multi_valued`). The walker's
`_predicate_is_functional` extends to consult the new value when
checking object_conflict. Phase 10.5 measures the impact on
derivation_corpus.

**Why deferred from v0.15.** Cluster 3's surface-form-canonicalization
and walker-promotion-shadow-prior work was already a focused scope
expansion; adding a new cardinality class is a third architectural
change with multi-corpus ripple. The cleanest sequence is v0.15
landing the in-place fixes (Cluster 3 step 7+8) with employed_by
left as multi-valued, then v0.16's predicate-metadata audit
reclassifying it alongside the related predicates as a coherent
set.


### D58 — TierU normalizer determinism between seed and promotion writes

**Surfaced by Phase H Cluster 3 step 7 fixup 2 (2026-05-26).** For
`der_revision_003` ("Asa is not a student" + prior polarity-1) and
`der_revision_004` (idempotent same claim + externally-verified
prior), the corpus runner seeds priors with the case's actual text
as source_text — matching the extractor's promotion path — so the
Wikipedia normalizer sees identical context for both writes.
Despite this, the seed's canonical subject differs from the
promotion's canonical subject (two Tier U rows result instead of
one for the idempotent case, and the polarity-conflict path doesn't
find the externally-verified prior for the polarity_conflict case).

The likely cause: `claim_id` differs ("seed" vs the extractor's
uuid) and the Wikipedia normalizer's LLM-based disambiguation
caches and/or hashes on claim_id. Even with identical
surface_form / source_text / claim_subject the LLM output can
differ run-to-run. The deeper fix would canonicalize seed and
promotion writes through a shared normalizer cache keyed on
surface_form only (not claim_id), OR write the seed AFTER the
promotion with the promotion's canonical form copied to the seed.

**Affected.** R4 cases that require the walker to bridge a seeded
externally-verified prior with a promoted in-session claim:
`der_revision_003` (polarity_conflict), `der_revision_004`
(idempotent). Both currently MISS in the seeded calibration mode
post-Cluster-3.

**Scope.** Audit the Wikipedia normalizer's caching / determinism
behavior (claim_id dependency, LLM call-site stability). Either
add a "canonical form lookup" path that's deterministic per
surface_form, or restructure the corpus runner to canonicalize
priors via the same code path that the extractor uses for its
claims.

**Priority.** Defer to v0.16. Cluster 3 closes with the +10pp lift
from the other R4 fixes (001, 005, 006). v0.16's D58 work plus the
already-captured D57 (functional-at-a-point-in-time) would close
the remaining R4 gap.

#### D56 refinement — predicate_distribution oracle never produces `both`

Phase 10.5 Session 1 surfaced that the predicate_distribution
oracle (Qwen 3-Next 80B with v2 prompt) never produces `both` as
a verdict across the `predicate_distribution_corpus`. All 5
`both`-category cases miss systematically in both seeded and
cold-start mode (since this corpus IS the cold-start oracle
test). The corpus passes the 85% threshold (88%, +3pp above) so
the systematic miss is not release-blocking, but the pattern is
worth investigating in v0.16.

**Three possible causes:**

1. **Prompt under-cues `both`.** The v2 prompt's framing may
   make `up` or `down` natural answers without `both` being
   surfaced as a distinct option. Prompt iteration could expand
   the `both` framing with examples and explicit non-trigger
   conditions per D45's positive-triggers + non-trigger pattern.

2. **Corpus `both` expectations are themselves debatable.** The
   5 predicates classified as `both` in the corpus may not have
   well-defined `both` semantics. An audit of which predicates
   were classified `both` and on what reasoning would clarify
   whether the cases are legitimate or aspirational.

3. **Model-specific limitation.** Qwen 3-Next 80B may not
   recognize `both` as a viable answer in the prompt's framing.
   An alternative model run (e.g., the candidates from the Phase
   E comparison) on the same `both` cases would distinguish a
   model-specific gap from a prompt-design gap.

**v0.16 work item.** Investigate which of the three causes (or
combination) explains the pattern:
- If prompt-tunable (cause 1): iterate the v2 prompt to surface
  `both` cases appropriately while preserving the current
  up/down/neither accuracy.
- If corpus-incorrect (cause 2): audit the corpus's `both`
  expectations and either justify or revise.
- If model-limited (cause 3): consider whether an alternative
  model handles the case better, or whether `both` is an
  oracle-frontier semantic that any model would struggle with.

Bounded investigation. Lands in v0.16 alongside other
predicate_distribution refinements. Surfaced by Phase 10.5
Session 1 calibration; documented in Session 2 Item 3.

### D59 — Claim model lacks `valid_from_ref` / `valid_until_ref` for event-relative bounds

**Surfaced by Phase 10.5 Session 2 temporal_scope diagnostic
(2026-05-28).** The `Claim` model (src/aedos/layer1_extraction/extractor.py)
carries `valid_from`, `valid_until`, and `valid_during_ref` as
temporal scope fields. It lacks `valid_from_ref` and
`valid_until_ref` for expressing event-relative bounds — "before
event X" or "after event Y".

The temporal_scope corpus's `ts_relative_007` and
`ts_relative_008` expected `valid_until_ref: claim_acquisition`
and `valid_from_ref: claim_election` respectively. The v0.15
Claim model can't carry these directly.

**v0.15 workaround (in place).** Extractor Rule 16 (added
2026-05-28, commit `586531f`) instructs the LLM to emit
`valid_during_ref` with a generated id for "before/after X event"
patterns. This is semantically loose — `valid_during_ref` means
"during X", but is also being used for "before/after X" — and the
runner can't distinguish the three event-relative patterns from
each other. The runner's `valid_from is None AND valid_until is None`
check passes regardless, because the load-bearing role of
`valid_during_ref` in v0.15 is to **suppress the implicit-past-tense
default** (`temporal.py:46` sets `valid_until='before_present'` for
past-tense verbs lacking other temporal signals).

**v0.16 work item.** Add `valid_from_ref` and `valid_until_ref`
fields to the `Claim` model. Update `extract_temporal_scope` in
`temporal.py` to honor them. Extend the corpus runner to check
the *_ref fields explicitly. Update the extractor prompt's Rule
16 to emit the semantically-correct field per the "before/after/
during" distinction. The v0.15 workaround can then be deprecated
cleanly: the loose `valid_during_ref` semantics get tightened
back to "during" only.

Related: the `valid_during_ref` value the LLM emits
("claim_acquisition", "claim_election", "claim_war", etc.) is
currently a generated id with no actual claim record backing it
in v0.15 — the extractor doesn't emit a separate claim for the
referenced event. v0.16 might also formalize whether
event-relative references require a backing event claim (Rule
9's `valid_during_ref` for "A when B" subordinate clauses DOES
emit B as a separate claim; Rule 16's event-relative bounds for
noun-phrase events do not). Consistency between Rule 9 and Rule
16 / valid_during_ref usage is a v0.16 follow-up.


### D60 — Calibration runner uses `claims[0]` only; ordering-sensitive on multi-claim outputs

**Surfaced by Phase 10.5 Session 2 variance pass + temporal_scope
diagnostic (2026-05-28).** The temporal_scope runner
(`_run_temporal_scope` in tests/calibration/test_corpus_runner.py)
reads `claims[0]` and compares its temporal fields against the
expected scope. For cases where the extractor produces multiple
claims (a decomposed extraction per Aedos's atomic-claim
discipline), only the first claim is scored — the rest are
silently dropped from the comparison.

This produced two distinct effects in Session 1 variance data:

- **ts_relative_005** ("The policy was in effect at the time of
  the merger") flipped seeded vs cold-start because the LLM
  produced two claims in some runs (the policy + the merger
  occurred), and which claim landed at index 0 depended on
  emission order.
- **ts_explicit_005** ("Einstein lived from 1879 to 1955") had
  variance pass / fail purely on extractor non-determinism with
  no underlying issue.

The diagnostic also surfaced that the runner's `claims[0]`
discipline doesn't validate the *full* expected behavior — for
`ts_explicit_010` ("X took effect on D1 and expired D2") the
corpus correction (commit `e92c72c`) expected the took_effect
claim at index 0; the expired claim is unverified by the runner
even though the extractor produces it.

**v0.16 work item.** Extend the runner's per-case comparison to:
- Accept an ordered list or set of expected claims per case
- Match each expected claim against any extracted claim (set
  semantics) rather than position-locked indexing
- Surface unmatched-extracted-claims as warnings (the extractor
  produced something the corpus didn't anticipate)

Reduces ordering sensitivity and lets the corpus test the full
decomposition shape rather than the first-claim shape. Out of
v0.15 scope; captured here for v0.16 alongside other
runner-discipline items (D24).
